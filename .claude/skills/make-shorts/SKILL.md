# make-shorts

## Working directory

Every command below is relative to this repo (`scripts/*.py`, `config.yaml`, `work/`). Claude Code's actual project root ‚Äî where it discovered this skill from ‚Äî may not be this repo (e.g. it was invoked from a parent directory's `.claude/skills/`). Before running anything below, confirm the shell's cwd is this repo's path; `cd` there first if it isn't.

## Prerequisites

Before the first run, make sure the environment is ready:

```bash
python scripts/setup.py
```

This checks/installs ffmpeg (via winget) and the Python dependencies, and reports whether a CUDA GPU was detected.

Make sure `config.yaml` exists (copy `config.example.yaml` if not) and read it before starting ‚Äî every step below is governed by it.

Speaker diarization (step 1c) is opt-in and not covered by `setup.py`: only needed when `config.diarization.enabled` is `true`. Requires `pip install pyannote.audio` plus a HuggingFace token (`HF_TOKEN` env var) that has accepted the gated model terms for `pyannote/speaker-diarization-3.1` and `pyannote/segmentation-3.0` on huggingface.co.

### First-ever run in this repo: offer the optional analysis features

Trigger this once: `<config.output_dir>/transcripts/` doesn't exist yet (i.e. no video has ever been processed here). Skip this whole section on every later run, even if the user declined everything the first time ‚Äî don't re-ask.

Before running the pipeline, tell the user in plain chat (not a wall of text) that three optional analysis features exist, each off by default and each purely additive (the pipeline works fine without them):

- **Speaker diarization** (`diarization.enabled`) ‚Äî labels who's talking per segment, feeds `coherence` scoring. Needs a free HuggingFace token, see README's "Optional: speaker diarization".
- **Audio-energy spike detection** (`audio_energy.enabled`) ‚Äî catches wordless hype moments (screams/laughs) transcript search misses. No token needed, just ffmpeg (already required).
- **Own-channel analytics grounding** (`scripts/youtube_analytics.py`, run separately from the pipeline) ‚Äî pulls this channel's real view/retention/traffic-source numbers into a local JSON so candidate-finding can weigh real performance instead of only general research. Needs a one-time Google OAuth setup, see README's "Grounding candidate-finding in your own channel's real performance".

Ask which (if any) they want set up now; for anything they decline, just proceed with it left off ‚Äî never block the actual clip-making on this. If they want diarization or audio-energy, flip the relevant `config.yaml` key(s) for them. If they want the YouTube analytics grounding, point them at the README section (it's an interactive one-time OAuth flow, not something to run unattended). Whatever they answer, continue straight into step 1 below afterward.

## Pipeline

Given a video path `<video>` and the loaded `config.yaml`:

### 1-2. Ingest (mechanical, one command)

Steps 1/1b/1c/1d/2 ‚Äî transcribe, pause detection, diarization, audio-energy spikes, chunking ‚Äî plus pre-extracting every chunk's visual-pass frames, are all deterministic given `config.yaml`: none of it needs judgment. Run them all in one call instead of invoking each script separately:

```bash
python scripts/ingest.py "<video>"
```

(pass `--config <path>` only if not using the repo's own `config.yaml`). Same pattern [work/_render.py](../../../work/_render.py) already uses for the render step, applied to the front half: it reads every gate/value from `config.yaml` itself, so nothing has to be threaded through by hand. It prints a JSON summary as its last line; read it for the paths the rest of the pipeline needs:

- `transcript_path` ‚Äî `<config.output_dir>/transcripts/<video_stem>.json`, produced by `scripts/transcribe.py`. Skipped (cached) if it already existed.
- `pauses_path` ‚Äî set only if `config.jumpcuts.enabled`; `<config.output_dir>/transcripts/<video_stem>_pauses.json` from `scripts/silence.py` (cached, adaptive FFmpeg `loudnorm` threshold ‚Äî not a guessed fixed dB value).
- `diarized` ‚Äî `true` only if `config.diarization.enabled` **and** `scripts/diarize.py` succeeded. It labels each transcript segment in place with a `"speaker"` field (`"–ì–æ–ª–æ—Å 1"`, `"–ì–æ–ª–æ—Å 2"`, ... by first appearance), idempotent. **Fails open**: if it errored (missing `HF_TOKEN`, `pyannote` not installed, 403/gated-repo, no GPU/OOM, etc.), `ingest.py` catches it, records a one-line warning, and continues as if diarization were off for this run ‚Äî tell the user that one line, then proceed; step 3 below just omits `coherence` from every candidate. Diarization needs `pip install pyannote.audio` plus an `HF_TOKEN` env var that has accepted the gated model terms for `pyannote/speaker-diarization-3.1` and `pyannote/segmentation-3.0` on huggingface.co.
- `energy_path` ‚Äî set only if `config.audio_energy.enabled`; `<config.output_dir>/transcripts/<video_stem>_energy_spikes.json` from `scripts/audio_energy.py` (cached). Catches sudden audio-energy jumps (screams, laughs, hype yells) relative to the file's own recent loudness ‚Äî a real scream/laugh/hype moment often transcribes to nothing meaningful or gets mangled by Whisper, so text-only candidate-finding can miss it outright; this pass exists to catch exactly that gap, not to replace the transcript search. **Fails open** the same way diarization does ‚Äî a warning is printed, the run continues with an empty spike list.
- `chunks_dir` / `chunk_paths` ‚Äî `work/<video_stem>/chunks/chunk_NNNN.json`, one per `analysis.chunk_minutes` window, from `scripts/chunker.py`.
- Frame pre-extraction (only if `config.visual.enabled`) ‚Äî for each chunk, sampled stills at `visual.frame_interval_seconds` are written to `work/<video_stem>/frames/chunk_NNNN/frame_*.jpg` via `scripts/frames.py`, skipped if that chunk's frames already exist. Step 3's visual pass below reads these directly ‚Äî do **not** re-run `scripts/frames.py` yourself for a chunk `ingest.py` already covered. **Fails open** per chunk: a failed extraction is recorded as a warning and does not block the rest of the run; report it, then proceed with step 3 for that chunk exactly as if `config.visual.enabled` were `false` for it (no stills to read).
- `warnings` ‚Äî every fail-open message above, collected; surface all of them to the user in one short line each before continuing to step 3.

If `ingest.py` itself exits non-zero (transcription or chunking failed ‚Äî the two non-optional steps), that's a hard stop: read the printed error and fix the underlying issue (ffmpeg/model/disk) before retrying, same as any other hard failure in this pipeline.

### 3. Find candidates (pass 1)

For each `chunk_NNNN.json` file, produce a `work/<video_stem>/candidates/candidates_chunk_NNNN.json` file containing a JSON list of objects `{"start": <seconds>, "end": <seconds>, "reason": "<short reason>", "coherence": <1-5, optional>}` for every strong moment found in that chunk's segments (jokes, reactions, stories ‚Äî judged from the text itself, not audio energy).

**Profanity/hype phrases are never a requirement for a candidate.** The actual bar is: would a viewer who has no other context find this genuinely interesting/entertaining ‚Äî a well-told story, a sharp observation, a clean joke, a surprising turn, a strong reaction ‚Äî judged from the coherence and content of the exchange itself, same as the `coherence` field measures continuity. A clean, calm, profanity-free exchange that lands a real point beats a loud/vulgar one that doesn't. Do not let the presence of `hype_phrases` or profanity substitute for this judgment call, and do not let their *absence* count against an otherwise strong moment.

**Don't gate candidates to "must-clip"/viral-tier strength (2026-07-23).** Surface genuinely interesting moments even when they're not obviously the strongest thing in the recording ‚Äî the channel is deliberately casting a wider net right now (freer hooks below, then reading real retention/reaction data on the results) instead of only banking the safest, biggest-looking bets. The one requirement that does NOT loosen: the moment still has to read as one coherent, complete piece of thought ‚Äî a story/exchange/reaction that starts and actually lands somewhere, not a random disjointed fragment grabbed because something was said. Judge this the same way the `coherence` field below defines it, even on a run where diarization is off and the field itself can't be set ‚Äî the judgment is qualitative either way: would a viewer follow one continuous idea to a real point, or does it just trail off/derail mid-thought. A modest, calm, low-hype moment with a clean line of thought is worth surfacing under this bar; a punchy-sounding fragment that never actually resolves is not.

Before scoring, read [docs/viral-clips-ru.md](docs/viral-clips-ru.md) ‚Äî it distills what actually makes a short-form clip get watched (hook-window placement, length sweet spot, single-arc vs. scattered structure, self-contained vs. needs-context) and apply it as a lens alongside the rules below: prefer a moment whose interesting part sits near the front (or can be trimmed to sit near the front in step 5) over one that needs a long setup, prefer a moment that resolves on its own over one that needs stream context the clip doesn't contain, and don't pad a candidate's window past where its content actually lands. This is a lens for judgment, not a hard filter ‚Äî a moment with a slightly buried hook is still worth surfacing if it's genuinely strong, just note the trim opportunity for step 5.

- **`coherence`** ‚Äî only set this field when the chunk's segments carry a `"speaker"` field (i.e. `config.diarization.enabled` was `true` for this run); omit it entirely otherwise. This measures how long the same train of thought is sustained across the candidate window, using the speaker labels to track it: does one speaker hold the floor developing a single idea/story/argument from start to an actual point, or does it derail ‚Äî cut off mid-thought, interrupted onto an unrelated tangent by another speaker, or hop between disconnected topics within the same window. Score 5 = one clean, sustained line of reasoning or story for the whole window, arriving somewhere; 3 = mostly one thread but with a minor detour or a slightly abrupt landing; 1 = disjointed ‚Äî several unrelated fragments stitched together, or cut off before the thought resolves. This is about topical/thematic continuity, not about how funny/strong the moment is ‚Äî a hilarious but scattered exchange still scores low, and a mild but tightly sustained train of thought scores high.
  - **Use this as an actual selection factor, not just a descriptive tag.** When a moment's line of reasoning clearly continues past the initially-noticed window (before or after), widen `start`/`end` (still within this chunk) to capture the full sustained thought rather than a smaller cut of it ‚Äî a longer, fully-developed train of thought is itself a reason to prefer this window over a shorter, choppier one covering the same beat. When two overlapping/nearby candidates compete for the same moment, prefer the version with higher `coherence`. Say so in `reason` when coherence is the deciding factor (e.g. `"reason": "sustained ~40s rant building to one punchline, no derailing"`).

- If `config.analysis.use_subagents` is `true` (default): dispatch one Agent (subagent_type: general-purpose) per chunk file, **in parallel in a single message**, each instructed to read its assigned chunk JSON and write its own `candidates_chunk_NNNN.json` file with that format. Do not have subagents talk to each other ‚Äî they work independently on disjoint time windows.
- If `config.analysis.use_subagents` is `false`: read every chunk file yourself, sequentially, and write the candidate files directly without dispatching agents.
- If `config.content.allow_mature` is `false`, instruct the search (subagent prompt or your own pass) to skip any moment that is primarily profanity or sexual/adult humor rather than including it ‚Äî only surface it as a candidate if it stands on its own without that material. If `true` (default), keep such moments as candidates normally; step 5 below flags them in the generated metadata instead of filtering them here.
- **Hard exclusion, independent of `allow_mature`** ‚Äî never surface a candidate whose "joke" is a slur targeting race/ethnicity/sexual orientation, or references child sexual abuse, even in jest among friends. `allow_mature` covers profanity and adult/sexual humor that stands on its own; it does not cover discriminatory or exploitative "jokes" ‚Äî those get skipped outright, no exception, regardless of how the moment otherwise scores.
- **Jargon-heaviness check** ‚Äî a moment that only lands if the viewer already knows this specific game's mechanics/vocabulary (a dense exchange of item names, ability combos, faction rules with no reaction/story/joke wrapped around it) is weaker than one that reads on its own to someone who's never played it, per [docs/viral-clips-ru.md](docs/viral-clips-ru.md)'s self-contained rule ‚Äî prefer the latter when both are otherwise similar strength. Dense mechanical chatter is fine as *texture* around a real reaction/punchline, but skip a candidate where the mechanical jargon IS the whole content.
- Treat any phrase in `config.analysis.hype_phrases` ‚Äî and other language in the same register (streamer/audience hype, meme call-outs, exaggerated reactions) ‚Äî as a tag that boosts a moment that already has real content (a joke, reaction, or story landed there). Do not manufacture a candidate purely because a hype phrase appears ‚Äî if the surrounding context is filler/nothing-actually-happened, skip it even with the phrase present. Exception: "–≥–æ –∫–ª–∏–ø"/"–∫–ª–∏–ø–∞–Ω–∏" is chat literally asking for a clip of this exact moment ‚Äî treat that as near-automatic inclusion rather than just a boost, still subject to the mature-content filter above.
- **Visual pass** ‚Äî only when `config.visual.enabled` is `true` (default `false`; the transcript-only search above still runs regardless). The game/topic can change partway through a recording, so this runs per chunk, not once for the whole video:
  1. Sampled stills for this chunk's time range are already sitting in `work/<video_stem>/frames/chunk_NNNN/frame_*.jpg` ‚Äî `scripts/ingest.py` (step 1-2 above) pre-extracted them for every chunk when `config.visual.enabled` is `true`. Only fall back to a manual `python scripts/frames.py "<video>" <chunk_start> <chunk_end> work/<video_stem>/frames/chunk_NNNN --interval-seconds <visual.frame_interval_seconds> --prefix frame` call if `ingest.py`'s summary flagged this chunk's extraction as a warning (failed open) or the directory is genuinely empty.
  2. Read the extracted `frame_*.jpg` files directly ‚Äî whichever agent is doing this chunk (the dispatched subagent from the bullet above, or yourself when `use_subagents` is `false`) looks at them alongside the transcript text it's already reading.
  3. If `config.visual.detect_game_context` is `true` (default): write a short game/topic label for this chunk to `work/<video_stem>/frames/chunk_NNNN/game_context.txt` (e.g. `Elden Ring`, `Valorant`, `Just Chatting`). Step 5 below reads it back in for any clip landing in this chunk's time range.
  4. If `config.visual.detect_visual_candidates` is `true` (default): add candidates for visually strong moments the transcript text alone wouldn't surface ‚Äî a death/game-over screen, a clutch play, a funny on-screen glitch, a big reaction ‚Äî to this chunk's `candidates_chunk_NNNN.json` list (same format as the text-based ones), with a `reason` that names what was actually seen, e.g. `"visual: death screen + chat spam"`, not a vague `"funny moment"`.
  5. **Cross-talk gate (mandatory, not a spot-check)** ‚Äî a multi-person voice call means someone can be talking about a *different* game/story than what's on screen (a friend recounting an earlier session, an unrelated tangent) while the recording still shows this chunk's game. Before adding ANY text-based candidate from this chunk, glance at one of the already-extracted `frame_*.jpg` stills inside its time window and confirm the dialogue topic actually matches `game_context.txt` (or matches "Just Chatting" content when that's the label). A candidate that fails this check is either dropped or, if still strong enough as pure banter unrelated to gameplay, kept but with `reason` stating the mismatch explicitly (e.g. `"reason": "banter about a different game (Dota) while footage shows HOI4 ‚Äî verified off-topic, kept anyway for the joke itself"`) so step 5 doesn't title/tag it against the wrong game. Skip this gate only when `config.visual.enabled` is `false` for the run (no frames exist to check against).
- **Audio energy pass** ‚Äî only when `config.audio_energy.enabled` is `true` (default `false`; requires step 1d). For this chunk's time range, take every entry from `<config.output_dir>/transcripts/<video_stem>_energy_spikes.json` whose `start`/`end` overlaps `[chunk_start, chunk_end]`. For each: check whether it already lines up with a candidate already found from the transcript/visual passes above (same or overlapping window) ‚Äî if so, skip it, the moment is already covered. Otherwise add it as its own candidate: read the transcript segments covering that window for context (there may be little or no real text ‚Äî that's expected, this pass exists for exactly that case) and write a `reason` that says what kind of spike it is if inferable from context (e.g. `"audio energy spike: scream reaction to death"`, `"audio energy spike: sudden laughter, no clear cause in transcript"`) rather than leaving the generic placeholder reason from the JSON file as-is.

Once every chunk has a candidates file, merge them:

```bash
python scripts/candidates.py work/<video_stem>/candidates work/<video_stem>/CANDIDATES.md work/<video_stem>/candidates.json
```

### 4. User approval

- If `config.analysis.require_approval` is `true` (default): show the user `work/<video_stem>/CANDIDATES.md` and ask which candidate IDs to proceed with. Only the approved subset continues to step 5.
- If `config.analysis.require_approval` is `false`: proceed with every candidate in `work/<video_stem>/candidates.json`.
- **Auto-select top-N cap** (Phase 10) ‚Äî apply this **only in auto-select mode**: when the user asked you to pick the best moments yourself ("—Å–¥–µ–ª–∞–π —Ç–æ–ø –º–æ–º–µ–Ω—Ç—ã –ø–æ —Ç–≤–æ–µ–º—É –≤—ã–±–æ—Ä—É" / "top moments, your call") rather than hand-listing candidate IDs. In that mode there is a **global** budget on how many standalone shorts to render from the whole recording, so a long stream doesn't dump dozens of clips:
  - Compute the budget from the **source recording length**, not this hour or this chunk: `python -c "from scripts.candidates import compute_moment_budget; print(compute_moment_budget(<source_duration_seconds>, <config.top_moments.rate_per_hour>, <config.top_moments.minimum>))"`. The default rate is 3/hour, so a 3-hour recording ‚áí 9 total ‚Äî chosen from anywhere across the 3 hours (all 9 may come from the last 10 minutes if those were the strongest), never 3-per-each-hour.
  - Assign each candidate a `score` from 1 (weak) to 5 (must-clip) using the same viral judgment as step 3 (reaction intensity, standalone-meme quality, audio-energy spike, chat-clip signals). Write the score onto each candidate row.
  - Select the survivors: `select_top_moments(candidates, budget)` keeps the highest-scored `budget` candidates (ties broken toward earlier moments) and returns them in timeline order. Only those continue to step 5. When the user hand-picks IDs, or asks for "all", skip this cap entirely ‚Äî it governs auto-select only.
- **Retention-informed scoring** (optional learning loop) ‚Äî before assigning the `score`s above, check for a retention-insights file at `work/_analytics/retention_insights.json` (produced by `scripts/retention.py`; absent on a fresh repo or when no clip has published-video data yet ‚Äî silently skip if missing). When present, let its channel-level signals nudge (never override) your judgment:
  - `by_tag` ranks how well past clips of each moment tag held viewers (`mean_average_watch`). If a candidate matches a tag that historically retains well, lean its score up; a tag that consistently bled viewers, lean it down.
  - `length_signal_counts` ‚Äî if the channel skews `too_long`, prefer tighter trims in step 5; if it skews `could_be_longer`, don't over-cut strong moments.
  - `common_cliff_zone` ‚Äî the recurring drop-off zone (as a fraction of clip length); design trims/hooks so the weakest stretch doesn't land there.
  This is a soft prior from the channel's own history, not a hard rule ‚Äî a clearly strong moment still wins regardless of its tag's past average.
  - **To (re)produce this file** (needs the one-time YouTube OAuth from the analytics-grounding setup): `python -m scripts.retention work/_analytics/retention_insights.json --markdown work/_analytics/retention.md --publish-queue work/_publish/queue.json` ‚Äî the insights JSON **must** be written to exactly `work/_analytics/retention_insights.json` (the path this bullet reads); the `--markdown` digest is for human review only. Re-run it after a batch of clips has accumulated real view/retention data to refresh the prior. Attribution into `by_tag` only fills in once published queue entries carry `moment_tag` / `source_duration` fields; without them the tag ranking stays empty but the length/cliff signals still work.

### 5. Refine (pass 2)

For each approved candidate, re-read that moment's transcript window (from the chunk file(s) covering its time range) and decide:

- **Exact trim points** ‚Äî adjust `start`/`end` to fall on natural speech pauses, not mid-word/mid-phrase, and keep the final duration within `config.clip.min_seconds`‚Äì`config.clip.max_seconds`. Per [docs/viral-clips-ru.md](docs/viral-clips-ru.md): if step 3 flagged (or you now notice) throat-clearing before the actual hook, move `start` past it so the interesting part plays near the very front ‚Äî don't keep filler setup just because it was inside the original candidate window.
- **Moment tag** (retention learning loop) ‚Äî assign **every** clip a short, free-form `moment_tag`: a few words naming the KIND of moment, in the same register as `reason` and never a fixed enum (e.g. `"—Ä–µ–∞–∫—Ü–∏—è –Ω–∞ —Å–º–µ—Ä—Ç—å"`, `"–∫–ª–∞—Ç—á"`, `"—Ç–∏–ª—å—Ç"`, `"—Å–º–µ—à–Ω–æ–π —Ñ–µ–π–ª"`, `"—á–∞—Ç –ø—Ä–æ—Å–∏—Ç –∫–ª–∏–ø"`). Pick whatever best captures the moment type. It is written into the clip's `PLAN.json` entry (below) and carried through the publish queue (via `enqueue`'s `moment_tag`), so `scripts/retention.py` can rank which KINDS of moments hold viewers (`by_tag`) and feed that back into step 4's scoring next run. Set it on compilation entries too (the compilation's dominant moment type). A `sub_threshold` clip's compilation `tag` (next bullet) may reuse the same phrase ‚Äî they describe the same thing. This field only powers the learning loop; it never affects rendering or the publish schedule, so a rough tag is fine ‚Äî but always set one.
- **Sub-threshold detection + tagging** (COMP-01, D-01) ‚Äî once the trim above is decided, check the resulting duration against `config.clip.min_seconds`. If it's still below that floor even at the tightest reasonable trim, do **not** force-pad it out to `min_seconds` ‚Äî instead first ask: is this a **standalone meme** ‚Äî an instantly-landing punchline/reaction that works with zero stream context (chat literally asking to clip it, an exploding reaction, a quotable one-liner) and runs at least ~6s at its tightest trim? If yes, mark it `standalone_meme: true` instead of `sub_threshold` and let it proceed through the rest of step 5 as a normal solo clip despite being under `min_seconds` ‚Äî the 2026-07 niche reference analysis found 6-20s meme-moments are the single most viral clip class, and folding one into a compilation buries it. This is a high bar: a merely-decent short moment is NOT a standalone meme. Otherwise mark this candidate `sub_threshold` and assign it a short, free-form `tag` describing the gameplay situation or theme (same register as `reason`, e.g. `"died to same boss"`, `"chat spam reaction"` ‚Äî never a fixed enum/category, this is a description for a later semantic-similarity judgment, not a lookup key). A sub-threshold candidate skips the rest of this step's per-candidate finishing work for this run ‚Äî crop_style, jump cuts, transitions, punch-zoom, subtitles, title/filename, and metadata are decided **once for the whole compilation** it may join in step 5b below (D-06), not per individual sub-threshold candidate. A candidate whose tightest trim already reaches `min_seconds` is completely unaffected by this bullet and proceeds through the rest of step 5 exactly as today.
  - Exception: **jump-cut computation** (the "Jump cuts" bullet below, gated on `config.jumpcuts.enabled`) still runs normally for a sub-threshold candidate if it applies ‚Äî removing dead air is orthogonal to whether the candidate ends up standalone or folded into a compilation, and its resulting `keep_segments` (if any) travels with the candidate into step 5b.
- **Crop style** ‚Äî determined by `config.crop.mode`:
  - If `config.crop.mode` is a concrete value (`zoom`, `pad`, or `original-16:9`), use that same value for every clip in this run ‚Äî do not vary it per clip.
  - If `config.crop.mode` is `auto`, choose per clip among all three documented options: `zoom` (visually dynamic moment, crop in tight), `pad` (dialogue/joke-driven moment where the visual matters less, reserve room for captions), or `original-16:9` (a moment where the full frame matters and cropping would cut off something relevant ‚Äî keep the whole 16:9 frame, letterboxed).
  - Never write `auto` into `PLAN.json` ‚Äî it must always be a concrete resolved value.
  - **Crop focus** ‚Äî only relevant when `crop_style` is `zoom` (the only style that actually crops horizontally; `pad`/`original-16:9` scale to full width, nothing to offset). The crop defaults to dead-center. If this clip's frames (from step 3's visual pass, or a fresh look via `frames.py` if needed) show the important content sitting off-center ‚Äî the player/action skewed to one side, a HUD element (minimap, kill-feed) pulling visual weight to the opposite side ‚Äî set `crop_focus_x` in the plan entry: `0` = crop hugs the left edge of the source frame, `100` = hugs the right edge, `50` (or omit) = center, values between interpolate. Judge this from the actual stills, not a guess ‚Äî omit the field entirely when the frame is already visually balanced or the moment isn't `zoom`.
- If `config.facecam.enabled` is `true`, let the facecam overlay inform (not override) the crop_style choice above ‚Äî e.g. prefer `pad` or `original-16:9` so the overlay stays in frame, rather than a `zoom` that might crop it out. `facecam.mode` and `facecam.region` describe where the overlay sits, for this judgment call only ‚Äî `render.py` does not crop to those pixel coordinates; it only ever applies the resolved crop_style (a center crop for `zoom`, or a full-frame letterbox for `pad`/`original-16:9`).
- **Jump cuts** ‚Äî only when `config.jumpcuts.enabled` is `true` (default `false`; requires step 1b's pause file):
  ```bash
  python scripts/jumpcuts.py keep-segments "<config.output_dir>/transcripts/<video_stem>_pauses.json" <start> <end> work/<video_stem>/jumpcuts/<clip_filename_stem>_keep.json --max-pause-seconds <jumpcuts.cut_threshold_seconds>
  ```
  `<start>`/`<end>` are this clip's trim points from the step above (absolute source seconds). This writes the sub-segments to actually keep after cutting out any pause longer than `jumpcuts.cut_threshold_seconds`.

  **Visual safety gate on gameplay footage (mandatory when it applies, not a spot-check)** ‚Äî a jump cut is audio-only: it has no idea whether a silent pause is genuinely dead air or a tense, wordless gameplay beat (a stealth setup, an ambush, someone lining up a shot, a HUD/kill/reveal moment) that a viewer needs to *see*, not just hear speech resume around. Only run this gate when `config.visual.enabled` is `true` **and** this clip's chunk's `work/<video_stem>/frames/chunk_NNNN/game_context.txt` names an actual game/activity, not `Just Chatting` ‚Äî a chatting-only chunk has no gameplay footage that a jump cut could wrongly eat, so plain audio-only cutting stays fine there. When the gate applies:
  1. List the written file's cut windows: `python scripts/jumpcuts.py boundary-windows work/<video_stem>/jumpcuts/<clip_filename_stem>_keep.json` ‚Äî prints `[[gap_start, gap_end], ...]`, one entry per boundary, in the same order as the file's segments.

This probes the source video once, then renders every entry in `PLAN.json` into `config.output_dir`, printing each output path. `render.py` also renders `"type": "compilation"` entries (step 5b) automatically via this exact same invocation ‚Äî no new CLI flags are required for compilations. When a plan entry has `keep_segments`, that clip is trimmed and concatenated from those sub-segments (the jump cuts from step 5) instead of a single continuous `start`‚Äì`end`; every other flag below still applies on top of the spliced result. Each clip's audio is optionally cleaned up (in this order): noise reduction (`--denoise`, FFmpeg `afftdn`, strength set by `--denoise-strength` in dB ‚Äî default `6`, gentler than FFmpeg's own default of `12`, because on a mixed game+voice track a strong setting reads as a wind-like/musical-noise artifact smeared across the non-voice audio while voice survives; raise it for a genuinely hissy mic, lower it further if game audio still sounds smeared) to strip mic hiss/hum, then loudness normalization (`--loudnorm`, FFmpeg `loudnorm`) to even out volume across clips, then the profanity mask (a plan entry's `profanity_spans`, if set, gets ducked+garbled via `--profanity-*` ‚Äî audio keeps flowing under the mask, it is not a silence cut), then the fade. Each clip's video can also get a cinematic pass: `--vignette` darkens the frame edges, `--grain-strength` (0-100, 0 = off) adds film grain, and a plan entry's `punch_zoom_at` (if set) triggers a snap-zoom-in sized by `--punch-zoom-amount` and paced by `--punch-zoom-ramp`. When `--emphasis-enabled` is passed, a plan entry's `emphasis_moves` (if set) add transient mid-clip zoom pulses on top ‚Äî each ramps in and back out (sized by `--emphasis-zoom-amount`, paced by `--emphasis-ramp`/`--emphasis-min-hold`, capped by `--emphasis-max-moves`), unlike `punch_zoom_at`'s hold-till-end zoom, and work on any crop style. Each clip also fades video and audio to black/silence over the last `config.clip.fade_seconds` seconds (default 0.5s) ‚Äî the fade only starts once the last word has fully finished (extending a bit into unused source footage past the clip's end when available, for non-jump-cut clips), it does not overlap or cut into speech. When a clip has `subtitles_path` set, the `--sub-*` flags style the burned-in captions (font/size/colors/position); the `--sub-position bottom` default keeps a safe margin from the very bottom of the frame so captions don't sit under TikTok/Reels/Shorts' own UI buttons. Add `--censor-subtitles` **only when `config.subtitles.censor_profanity` is `true`** (store-true, omit when `false`) to partially mask profane words in the burned-in captions ‚Äî it keeps a leading prefix and replaces the rest with `*` (e.g. `—Ö—É–π` ‚Üí `—Ö**`), so the word stays recognizable without being spelled out. This is the on-screen text counterpart to the audio profanity mask and is independently toggleable (a clip can bleep audio without masking text, or vice versa). When on, also pass `--censor-keep-ratio <config.subtitles.censor_keep_ratio>` and `--profanity-wordlist <config.profanity.wordlist_path>` (reuse the same wordlist as the audio mask so both layers hide the same words); fail-open ‚Äî a missing/malformed wordlist masks nothing rather than blocking the render.

## Real channel performance (optional)

If `<config.output_dir>/analytics/channel_performance.json` exists (produced by `python scripts/youtube_analytics.py`, see README ‚Äî requires one-time OAuth setup, not run automatically as part of this skill), read it before step 3 when it's present and treat it the same way as [docs/viral-clips-ru.md](docs/viral-clips-ru.md): a lens, not a hard filter. Don't fetch or refresh this file yourself as part of the pipeline ‚Äî it's a manually-run, occasional snapshot; just read it if it's already there.
