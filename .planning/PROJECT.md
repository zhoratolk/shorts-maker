# shorts-maker

## What This Is

Локальный пайплайн, превращающий длинную запись геймплейного стрима в вертикальные (9:16) шортсы: Whisper-транскрипция, семантический поиск залипательных моментов, монтаж (jumpcuts, punch-zoom), без вотермарок и лимитов по времени. Для одного стримера/канала, запускается локально через Claude Code (`/make-shorts <путь-к-видео>`).

## Core Value

Из сырой записи стрима автоматически получить готовый к публикации вертикальный клип — без ручной нарезки и без потери самых залипательных моментов.

## Requirements

### Validated

- ✓ Транскрипция звука через faster-whisper (кэшируется, детерминирована) — существующий MVP
- ✓ Поиск кандидатов-моментов через семантический анализ чанков транскрипта (параллельные Task-сабагенты) — существующий MVP
- ✓ Опциональные сигналы: спикер-диаризация (pyannote), audio-energy спайки — существующий MVP, off by default, fail-open
- ✓ Монтаж: jumpcuts (вырезание пауз), punch-zoom на payoff — существующий MVP
- ✓ Вертикальный формат 9:16 без вотермарок и лимита длины — существующий MVP
- ✓ Grounding в реальной статистике канала через `scripts/youtube_analytics.py` (OAuth, YouTube Analytics API, fail-open) — существующий MVP
- ✓ Личные/канальные данные никогда не попадают в репозиторий/историю git (см. Key Decisions) — подтверждено сессией 2026-07-07
- ✓ **Компиляция недостаточных моментов** — моменты короче `config.clip.min_seconds` тегируются вместо отбрасывания (COMP-01), похожие по геймплейной ситуации/теме группируются в рамках одного video_stem (COMP-03) и стыкуются через переходный движок Фазы 4 в один полноценный шортс (COMP-02) — Фаза 5, включая gap-closure план 05-05 (исправлен критический баг CR-01: несогласованность pre-cap/post-cap длины `boundary_transitions` в `build_compilation_entry`, блокировавший COMP-02)
- ✓ **Авто-паблиш на TikTok и Instagram Reels** — то же расписание/dry-run/pause-kill/idempotency, что и YouTube (Фаза 3), распространено на TikTok через Content Posting API (PUB-06) и Instagram через Graph API (PUB-07); все три площадки изолированы друг от друга (Success Criterion 3, доказано выделенными изоляционными тестами) — Фаза 6, включая найденный на верификации и исправленный критический баг CR-01 (сигнал SELF_ONLY/pre-audit не сохранялся на пути восстановления после сбоя `reconcile_uploading`, из-за чего пост мог молча пометиться как опубликованный, даже оставшись приватным; исправлено коммитом 71a6131, повторная верификация — 11/11)

### Active

- [ ] **Лимиты монетизации по площадкам** — анализ правил YouTube Shorts, TikTok и Instagram Reels (copyright, hype speech, gambling-темы, гейминг-специфика и т.д.) и пометка в метаданных клипа мест, где монетизацию могут снять
- [ ] **Анализ выбора и названий своих шортсов** — используя историю из `scripts/youtube_analytics.py` (реальные загруженные видео канала), вывести паттерны: какие моменты выбирались, как называются — чтобы новые кандидаты/заголовки были стилистически похожи
- [ ] **LLM-генерация тегов и названий** — пайплайн сам анализирует видео/транскрипт и предлагает готовые варианты названия + список тегов на кандидата (не просто сигналы для ручного заголовка)
- [ ] **Контекстные переходы между клипами** — новый набор типов переходов помимо cut/punch-zoom: match cut, whip pan, mask/wipe, glitch, crossfade. Выбор типа перехода — по контексту сцены/аудио на стыке двух клипов (анализ движения/громкости в конце клипа A и начале клипа B), не фиксированный набор

### Out of Scope

- Заливка через веб-интерфейс каждой площадки вручную — заменяется авто-паблишем по API (фаза 6)
- Поддержка нескольких стримеров/каналов одновременно — пайплайн заточен под один канал пользователя
- Полностью новый видеоредактор — только программные эффекты поверх moviepy/ffmpeg, без GUI

## Context

Проект уже прошёл через MVP-фазу (транскрипция → поиск моментов → монтаж → базовые эффекты) и через инцидент безопасности: коммит с реальной статистикой канала (просмотры/названия конкретных клипов) был опубликован в `docs/viral-clips-ru.md`, затем полностью вычищен из git-истории (`git rebase` drop + `push --force-with-lease`) по прямому запросу пользователя. Урок закреплён явным требованием: **личные/канальные данные пользователя не должны существовать в репозитории** — ни в коде, ни в документации, ни в git-истории. Кэш реальной статистики канала (`scripts/youtube_analytics.py`) обязан оставаться gitignored.

Кодовая база смэплена в `.planning/codebase/` (STACK, ARCHITECTURE, STRUCTURE, CONVENTIONS, TESTING, INTEGRATIONS, CONCERNS) — брать оттуда контекст перед планированием новых фаз.

## Constraints

- **Приватность**: реальные данные канала (статистика, конкретные названия/просмотры видео) никогда не коммитятся — только через gitignored кэш. Обязательно к соблюдению во всех новых фазах, особенно в фазе анализа названий (фаза 2) и авто-паблиша (фаза 6)
- **Локальность**: весь пайплайн работает on-device (Whisper локально, ffmpeg локально) — новые фазы (LLM-теги, авто-паблиш) не должны требовать постоянного облачного бэкенда, кроме уже используемых API самих площадок
- **Fail-open**: опциональные фичи (диаризация, audio-energy, YouTube Analytics grounding) обязаны деградировать без падения пайплайна при недоступности — тот же принцип должен распространяться на новые опциональные фичи (например, LLM-теги при недоступности сети)
- **Git-дисциплина**: коммитить можно свободно по ходу работы; пуш и любые деструктивные git-операции — только по явному запросу пользователя в каждом отдельном случае
- **Необратимость авто-паблиша**: публикация на площадки — публичное и трудно отменяемое действие; фаза 6 должна включать защитные механизмы (dry-run режим, возможность приостановить расписание) — это не одноразовое решение "запушить", а встроенная безопасность в саму фичу

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| История git переписана (drop commit + force-push) для удаления реальных данных канала | Пользователь явно запросил после того, как коммит с личной статистикой оказался уже запушен | ✓ Good |
| Описание скилла `make-shorts` дополнено явной "Use when..." триггер-фразой | В новых сессиях того же проекта скилл срабатывал только на буквальный `/make-shorts`, не на естественные формулировки типа "нарежь видео" | ✓ Good |
| Новые переходы выбираются по контексту (анализ сцены/аудио), а не фиксированным набором | Пользователь предпочёл более умный, хоть и более сложный в реализации вариант | — Pending |
| Компиляция недостаточных моментов — отдельная фаза с кластеризацией по похожести (геймплей/шутка), а не просто "склеить что осталось" | Пользователь настоял на смысловой связности итогового ролика | ✓ Good — Фаза 5 завершена (включая gap-closure 05-05) |

## Evolution

Этот документ развивается на переходах между фазами и на границах milestone.

**После каждого перехода фазы** (через `/gsd-transition`):
1. Требования инвалидированы? → перенести в Out of Scope с причиной
2. Требования подтверждены? → перенести в Validated со ссылкой на фазу
3. Появились новые требования? → добавить в Active
4. Есть решения для лога? → добавить в Key Decisions
5. "What This Is" всё ещё точно? → обновить при расхождении

**После каждого milestone** (через `/gsd-complete-milestone`):
1. Полный обзор всех секций
2. Проверка Core Value — всё ещё правильный приоритет?
3. Аудит Out of Scope — причины всё ещё валидны?
4. Обновить Context текущим состоянием
