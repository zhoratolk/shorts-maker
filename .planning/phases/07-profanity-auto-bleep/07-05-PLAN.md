<objective>
The single manual gate this phase needs: a human listens to one real masked render and confirms the duck+garble parameter values sound right, or retunes them in config.

Purpose: AUDIO-03 / D-03 demand the mask be "quiet/garbled enough to defeat STT without being so loud/abrupt that it reads as an obvious edit." The loudness-delta and self-transcription tests (Plan 07-04) prove the mask ducks and defeats STT, but "doesn't sound like a jarring cut" is a subjective audio-quality judgment with no full automated proxy — D-03 and 07-RESEARCH Open Question 2 both explicitly call for empirical validation against a real clip. The duck/garble constants (07-RESEARCH Assumptions Log A1) are documented starting points; this checkpoint confirms or adjusts them.
Output: human sign-off (or a config retune committed to config.yaml), recorded in the SUMMARY.
</objective>

  <action>
    This is a human-verification checkpoint, not an automated task — do not attempt to auto-approve it (subjective audio quality has no automated proxy). Automate everything up to the point of listening: ensure config.profanity.enabled can be set true and a masked clip can be rendered (per 07-04-SUMMARY.md), then hand off to the human with the steps below. On "approved", record the final duck/garble values in the SUMMARY; on a retune request, adjust config.profanity values and re-render before re-presenting.
  </action>
  <what-built>
    The full profanity auto-bleep feature (Plans 07-01…07-04): deterministic RU+EN detection over the Whisper transcript, a duck+garble ffmpeg mask applied only inside detected swear spans, a fail-open config toggle, and SKILL.md orchestration producing `profanity_spans` in PLAN.json. All automated checks (unit + real-ffmpeg loudness-delta + faster-whisper self-transcription) are green.
  </what-built>
  <how-to-verify>
    1. Pick a source recording that contains at least one clear swear word (RU and/or EN).
    2. In `config.yaml`, set `profanity.enabled: true` (copy the `profanity:` section from `config.example.yaml` if absent). Leave the duck/garble defaults for the first pass.
    3. Run the pipeline for that video (`/make-shorts <path>`) through to render — or, faster, render one clip directly with `scripts/render.py` on a PLAN.json entry that includes a `profanity_spans` covering the known swear word (see 07-04-SUMMARY.md for the exact invocation).
    4. Play the rendered clip and listen at each masked word. Confirm ALL of:
       - the profanity is hard to make out (quiet + garbled), not cleanly intelligible;
       - the audio keeps flowing underneath — no dead-silence gap (AUDIO-02);
       - it does NOT read as an abrupt hard cut / obvious edit (D-03).
    5. If any of those fail, adjust in `config.yaml`'s `profanity:` section and re-render: raise/lower `duck_volume` (lower = quieter word), widen/narrow `garble_width_octaves` / shift `garble_freq` (STT-defeat vs. naturalness), tune `warble_freq`/`warble_depth` (garble texture). Repeat until satisfied. Record the final chosen values in the SUMMARY.
  </how-to-verify>
  <resume-signal>Type "approved" (optionally noting the final duck/garble values used), or describe what still sounds wrong so the parameters can be retuned.</resume-signal>
