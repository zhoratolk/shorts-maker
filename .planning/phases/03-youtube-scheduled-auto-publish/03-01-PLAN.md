<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local queue.json → publish logic | The manifest is local-only and not attacker-controlled in the classic sense, but a corrupted/hand-edited entry (bad status, missing field) must fail loud rather than silently drive a wrong publish later — treat as trusted-but-verify. |
| config.yaml → publish behavior | publish.enabled is the single switch between dry-run and live; its default and validation are safety-critical (a mis-default to true would enable live uploads without opt-in). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | enqueue() sequential numbering | low | mitigate | seq derived solely from persisted max(seq)+1 and idempotent on clip_id — a duplicate enqueue can't silently renumber or double-queue a clip (unit-tested this plan). |
| T-03-02 | Elevation of Privilege | publish.enabled default | high | mitigate | Dataclass default enabled=False AND a missing `publish` section resolves to enabled=False; both are asserted by tests so dry-run can never regress to live-by-default (PUB-03). |
| T-03-03 | Denial of Service | load_queue on missing/corrupt file | low | mitigate | load_queue returns {"entries": []} for a missing file (fail-open, project convention); malformed JSON surfaces as a hard load error rather than a silent empty queue that would re-process everything. |
</threat_model>
