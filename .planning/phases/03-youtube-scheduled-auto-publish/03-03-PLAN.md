<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kill request → YouTube Data API | A kill must actually prevent a public post; a silently-failed kill is a safety failure (unwanted public content). |
| stuck manifest entry → retry logic | The gap between "upload started" and "upload confirmed"; the wrong resolution here double-posts publicly. |
| channel uploads list → seq-marker match | Matching a YouTube video back to a local entry relies on the embedded marker being specific and unforgeable within this channel's own content. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-08 | Repudiation / silent failure | kill of a scheduled video | high | mitigate | kill_item marks KILLED only after verify_killed re-fetches status and confirms private (Pitfall 2); a failed verify surfaces loudly and blocks the KILLED mark. Plus the end-of-phase human-check empirically confirms Assumption A1 before the kill path is trusted live. |
| T-03-09 | Tampering / duplicate post | reconcile_uploading match | high | mitigate | Reconciliation matches the specific seq marker embedded in the description before deciding adopt-vs-requeue; no match → requeue only (safe, nothing was created); match → adopt existing video_id, never re-insert (PUB-05). Reconciliation runs before any new selection so a stuck entry can't be bypassed. |
| T-03-10 | Denial of Service | reconcile network failure | medium | mitigate | If the uploads-list call fails (network), reconciliation logs-and-leaves the entry UPLOADING rather than guessing — it does NOT reset to QUEUED on an inconclusive check (which could later double-upload). Fail-safe toward not-retrying, matching the project's fail-open-but-never-destructive posture. |
</threat_model>
