<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Task Scheduler → publish_queue.py --check | A detached background process runs the upload path with the user's own credentials; a misconfigured (duplicate/too-frequent) trigger could over-run the queue or quota. |
| notification log → chat surfacing | The only path by which a session-less run's outcome reaches the user; a lost or double-counted line is a visibility failure, not a safety one. |
| CLI args → publish action | --now/--kill act on a live channel; an unknown/typo'd clip_id must error, not act on the wrong item. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-11 | Denial of Service / quota | duplicate Task Scheduler entries | medium | mitigate | --check uploads at most one item per invocation (natural debounce), and docs/publish-queue.md includes the `schtasks /query` verification that exactly one task entry exists (Pitfall 4). |
| T-03-12 | Elevation of Privilege | --check honoring dry-run | high | mitigate | The --check dispatch reads publish.enabled and short-circuits to a logged skip when false — same gate as Plan 02, re-asserted at the CLI seam and unit-tested (PUB-03). |
| T-03-13 | Repudiation | lost notification on session-less run | low | mitigate | append-only log + last-read marker guarantees a session-less run's outcome is durably recorded and reported exactly once at the next session (D-06). |
| T-03-14 | Tampering | --now/--kill targeting wrong clip | medium | mitigate | Unknown clip_id errors cleanly (parser.error / no-op) rather than acting on a default/adjacent item; tested. |
</threat_model>
